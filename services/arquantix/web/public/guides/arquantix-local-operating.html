Le plus simple au quotidien

Si vous ne savez pas quoi faire

Une seule commande, la plus sûre pour remettre le projet en état sans rien casser (pas de suppression de base, pas de volumes effacés) :

make doctor-fix

Elle remet les conteneurs au propre (redémarrage ciblé si besoin). Pour le détail, voir la section Diagnostic.

Si tout va bien après le démarrage de votre Mac, vous n’avez souvent qu’à vérifier que l’application répond. Inutile de tout relancer « au cas où » : cela peut même ralentir votre travail.

Adresses utiles (ports par défaut ; les vôtres peuvent différer si le fichier .env.arquantix dit autre chose) :

  • API (santé) — http://127.0.0.1:8000/health
  • Web (interface admin) — http://127.0.0.1:3000/

Toutes les commandes ci-dessous se lancent depuis la racine du dossier du projet (là où se trouve le Makefile).

Vérification rapide recommandée — un tableau de bord lisible dans le terminal :

make status

Lecture seule : rien n’est modifié sur votre machine ni sur Docker.

make doctor-fix ou make setup ?

Commande Quand l’utiliser
make doctor-fix Quelque chose cloche (service, page blanche, API qui ne répond plus), ou après une modif de code : remise en route sûre, sans tout réinstaller.
make setup Première installation, gros changement dans .env.arquantix, ou vous voulez tout relancer proprement après un long arrêt / un changement d’environnement (en gardant vos données).

À retenir : en cas de doute sur un bug ou un service à « réparer », commencez par make doctor-fix. Gardez make setup pour les situations plus larges (install, gros changement de config).

Surveillance en direct : make status-watch = suivi local en temps réel, comme un mini tableau de bord live dans le terminal (rafraîchissement automatique ; Ctrl+C pour arrêter).

État attendu quand tout marche

Vous pouvez cocher mentalement ces trois points :

  • API — l’adresse /health répond (souvent une réponse « OK » ou JSON, pas une erreur).
  • Web admin — la page d’accueil charge sans erreur bloquante (ex. http://127.0.0.1:3000/).
  • Terminal — make status affiche les services en vert / OK, sans erreurs critiques.

Si l’un de ces trois éléments échoue, enchaînez avec make doctor puis, si besoin, make doctor-fix.

Quand j’allume mon Mac

Normalement :

  1. Docker Desktop démarre (souvent avec la session).
  2. Si vous avez installé l’autostart (LaunchAgent), la stack officielle peut se lancer automatiquement au login — voir docs/LOCAL_MAC_AUTOSTART.md.
  3. Sinon, vous pouvez lancer vous-même une remise en route propre (voir ci-dessous).

Option A — Regarder l’état (sans rien changer)

make status

Idéal pour voir d’un coup d’œil si les services tournent et si l’API / le web répondent.

Option B — Relancer proprement la stack recovery (recommandé si Docker était arrêté ou si rien ne répond)

Script prévu pour le démarrage / login : il attend que Docker soit prêt, puis lance la stack sans supprimer les données.

bash scripts/start_arquantix_recovery_boot.sh

Équivalent documenté côté Makefile : make -f Makefile.arquantix arquantix-recovery-up — même famille de commandes, même projet officiel.

Option C — Forcer le LaunchAgent à rejouer son script (Mac avec autostart installé)

À utiliser si l’agent est chargé mais que vous voulez déclencher à nouveau le même démarrage que lors du login :

launchctl kickstart -k gui/$(id -u)/com.arquantix.autostart
Quand laquelle choisir ?
  • Doute ou contrôle → make status d’abord.
  • Stack arrêtée ou après réveil du Mac → start_arquantix_recovery_boot.sh (ou autostart au prochain login).
  • Autostart installé mais pas relancé → kickstart sur le label com.arquantix.autostart.

Quand j’ai modifié du code et je veux repartir proprement

Ici, « repartir proprement » veut dire : les conteneurs voient vos derniers changements ou sont redémarrés, sans effacer la base de données.

Backend (API Python)

Souvent, un redémarrage du service API suffit après une modif. Le moyen sûr et intégré au projet :

make doctor-fix

Fait un compose up idempotent puis, si besoin, redémarre seulement arquantix-api ou arquantix-web. Jamais de down -v.

Si vous avez changé des fichiers Docker (Dockerfile, dépendances système), il peut falloir une reconstruction d’images — voir avec l’équipe ou make -f Makefile.arquantix arquantix-build (cas avancé).

Web admin (Next.js)

Même logique : make doctor-fix peut redémarrer le service web si l’API est OK mais pas l’interface. Sinon, un simple rechargement de la page suffit souvent.

Configuration (.env, ports)

Après changement dans .env.arquantix, il faut que Docker relise la config :

make setup

make setup relance la stack proprement (volumes conservés) et attend que l’API réponde sur /health. C’est la voie simple pour « tout remettre d’aplomb » après un changement d’environnement.

Quand un simple test suffit

Si vous n’avez touché qu’au code déjà pris en charge par le rechargement à chaud dans le conteneur, ouvrez simplement :

  • /health pour l’API
  • / pour le web
Rappel : make doctor-fix est pensé pour corriger sans casser les données. make setup est utile pour une remise en route complète après install ou gros changement de config.

Quand je veux « repartir à zéro » sans casser la base

Dans ce guide, « repartir à zéro » signifie : redémarrer les services, recharger le code, retrouver un état cohérent — sans supprimer vos données PostgreSQL locales.

Commande simple et sûre pour remettre la stack en route (données conservées) :

make setup

Si le souci est plutôt « un service coince » que « tout est à refaire » :

make doctor-fix

Ne jamais faire ceci (perte ou risque pour les données)

  • docker compose down -v — supprime les volumes liés, souvent la base.
  • docker volume prune — peut effacer des volumes non utilisés, dangereux si mal compris.
  • docker system prune --volumes — encore plus large et risqué.

Ne pas utiliser l’ancien projet Docker « arquantix » pour le quotidien — la stack officielle locale est arquantixrecovery + docker-compose.arquantix-recovery.yml.

Diagnostic : doctor, correctifs, tableau de bord

Toutes ces commandes sont lancées depuis la racine du dépôt.

Commande À quoi ça sert Quand l’utiliser Sécurité
make doctor Diagnostic détaillé (Docker, fichiers, services, HTTP, base, Redis…). Affiche un verdict et une suggestion si problème grave. Dès que quelque chose semble « cassé » ou avant une démo. Lecture seule Ne modifie rien.
make doctor-fix Actions de remise en route sûres : compose up idempotent, redémarrage ciblé API ou web si besoin. Après make doctor qui signale des erreurs, ou pour relancer proprement sans tout casser. Sûr Pas de down -v, pas de prune volumes.
DRY_RUN=1 make doctor-fix Même logique que doctor-fix, mais affiche seulement ce qui serait exécuté (simulation). Pour comprendre ce qui se passerait sans rien lancer. Simulation Aucune action Docker réelle.
make status Tableau de bord texte : projet compose, services, ports, santé API/web/DB/Redis, alertes (legacy…). Vue rapide « tout va bien ? ». Lecture seule
make status-watch Mini tableau de bord live dans le terminal : même contenu que make status, mis à jour en boucle (rafraîchissement régulier ; Ctrl+C pour arrêter). Intervalle optionnel : STATUS_REFRESH_SEC=5 make status-watch. Pour surveiller pendant un démarrage ou pendant que vous testez une modification. Lecture seule

Scénarios concrets

Mon Mac vient de démarrer

Ouvrir les URLs ou lancer make status. Si vide ou erreur, bash scripts/start_arquantix_recovery_boot.sh.

make status

L’API ne répond pas

Vérifier /health, puis make doctor. Ensuite make doctor-fix si besoin.

make doctor
make doctor-fix

Le web admin affiche une erreur

Vérifier l’API d’abord. Puis make doctor-fix (redémarrage web possible).

make doctor-fix

Je viens de modifier le backend

Tester /health. Si besoin : make doctor-fix.

make doctor-fix

Je veux vérifier que tout est sain

Diagnostic complet, sans modification :

make doctor

Je veux voir l’état en direct

Tableau qui se rafraîchit (quitter avec Ctrl+C) :

make status-watch

Je veux relancer sans rien casser

Remise en route douce : make doctor-fix. Remise en route complète avec attente API : make setup.

make doctor-fix

Checklist ultra simple

Chaque matin (ou après une pause longue)

  1. make status — coup d’œil sur les services.
  2. Ouvrir /health — l’API doit répondre.
  3. Ouvrir le web admin.

Après une modification de code ou de config

  1. Relancer proprement si besoin : make doctor-fix ou make setup (gros changement d’environnement).
  2. Vérifier l’API (/health).
  3. Retester l’interface web ou votre appli mobile (ex. Flutter) pointant sur la même API.

En cas de doute

  1. make doctor — comprendre le problème.
  2. DRY_RUN=1 make doctor-fix — voir ce qui serait fait.
  3. make doctor-fix — appliquer les correctifs sûrs.

Mots à ne pas oublier

  • Stack officielle locale — projet Compose : arquantixrecovery
  • Fichier Compose — docker-compose.arquantix-recovery.yml
  • Base locale utile (nom courant dans la doc) — arquantix_fresh (souvent dans .env.arquantix)
  • À éviter au quotidien — l’ancien namespace / projet arquantix (legacy), qui n’est pas la stack recommandée
  • Strapi / CMS — retiré du projet ; la stack utile est : arquantix-db, arquantix-redis, arquantix-api, arquantix-web
