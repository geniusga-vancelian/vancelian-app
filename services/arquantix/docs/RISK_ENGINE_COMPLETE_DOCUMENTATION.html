1. Introduction

Objectif du Risk Engine : enrichir la décision de sécurité après les politiques d’authentification statiques et la friction adaptative « simple », en produisant un score 0–100, un niveau (low / medium / high / critical), des facteurs pondérés explicables, et une recommandation (allow / step_up / reauth) — le tout sans modèle ML dans le moteur (risk_engine.py, docstring).

Pourquoi déterministe : auditabilité, répétabilité, explication par facteurs, absence de boîte noire ; les poids sont versionnés et surchargeables (risk_config.py).

Product implication

Les couches se cumulent : la policy fixe le niveau d’exigence par type d’action ; la friction adaptative (5B) peut réduire le step-up dans des cas bas risque avant le moteur ; le risk engine peut escalader (step-up / reauth) mais respecte une reauth déjà requise par la couche continue (continuous_auth_engine._merge_risk_into_decision).

CoucheRôle
Policy statique (sensitive_action_map)Niveaux LOW/MEDIUM/HIGH, step-up obligatoire, fenêtres recent auth, trusted-only — ne calcule pas de score métier.
Friction adaptative 5B (continuous_auth_engine)Règles montant / device / récence pour alléger certaines actions si flags activés.
Risk engine 5C (risk_engine.evaluate_request_risk)Somme pondérée + seuils + recommandation ; intègre facteurs session, montant, comportementaux si activés.
Behavioral risk 5DSignaux géo, appareil, rafales, etc. ; bornés par facteur (clamp_behavioral_weight : −20 … +40).
Adaptive intelligence 5ESegment utilisateur, ajustement de score, seuils high/critical par segment, seuils dynamiques pour friction (montant / récence) utilisés ailleurs.
Calibration / feedback 5FFeedback typé, suggestions de poids non automatiques, expériences A/B sur poids, version de calibration.

Ordre d’exécution réel (continuous_auth_engine.evaluate_request_security_context) :

  1. Si auth continue ou SI désactivés → décision permissive (continuous_auth_disabled / no_session_intelligence).
  2. Application des règles policy + session intelligence + friction adaptative 5B (éventuelle réduction de step-up).
  3. Si is_risk_engine_enabled() et action sensible : evaluate_request_risk(...) avec le contexte strict (reauth déjà requise, friction adaptative appliquée, etc.).
  4. Fusion : une require_reauth déjà vraie court-circuite l’escalade risque ; sinon application des outcomes risque (reauth / step_up).

Dashboard (risk_dashboard_store.py, routes /admin/security/risk-dashboard/*) : agrégations en mémoire par processus pour métriques produit — voir PHASE_5_DASHBOARD_PRODUCT.md.

3. Politiques statiques

Fichier sensitive_action_map.py : ACTION_POLICIES par action_key (SensitiveActionPolicy : required_auth_level, requires_step_up, requires_recent_auth_seconds, requires_biometric, allowed_if_device_trusted_only).

Ce que les policies définissent : exigences d’auth continue avant le score (via require_continuous_auth_for_action + moteur continu).

Ce qu’elles ne définissent pas : les poids du risk engine, les facteurs comportementaux, les segments 5E — ce sont des couches séparées.

5.3 Seuils de niveau (derive_risk_level)

Par défaut : critical si score ≥ critical_threshold (env RISK_CRITICAL_THRESHOLD, défaut 75) ; high si ≥ high_threshold (RISK_HIGH_THRESHOLD, défaut 50) ; medium si ≥ 25 ; sinon low.

Avec 5E, segment_risk_thresholds modifie les deux premiers seuils par segment (ex. trusted_user : 60 / 80 ; new_user / risky_user : 40 / 65).

Security implication

Les overrides durs behavioral_force_reauth fixent recommended_outcome à reauth et risk_level à critical (evaluate_request_risk). behavioral_force_step_up force au minimum step_up et remonte le niveau low → medium si besoin.

6. Règles de décision

Fonction _recommended_outcome : si strict_require_reauth (contexte continu) → reauth ; sinon selon niveau : critical → reauth ; high → step_up ; medium : si 5E et segment trusted_user → allow ; si risky_user → step_up ; sinon allow si friction adaptative appliquée, sinon step_up ; low → allow.

Priorité reauth stricte : dans _merge_risk_into_decision, si dec.require_reauth déjà vrai, le moteur de risque n’écrase pas cette exigence.

Escalade : recommandation risque reauth ou step_up peut augmenter la friction ; downgrade de clé métier : uniquement wallet_transfer → internal_transfer_low sous conditions (section 7).

7. internal_transfer_low

Pourquoi : permettre un reclassement d’action lorsque le transfert est manifestement bas risque (même titulaire, petit montant, appareil fiable, step-up récent, score < 25) et ADAPTIVE_FRICTION_ENABLED — aligné sur la doc interne Phase 5C.

Règles exactes (_downgrade_wallet_to_internal_transfer_low) — toutes doivent être vraies :

  • is_adaptive_friction_enabled()
  • action_key == "wallet_transfer"
  • amount_eur présent et < low_risk_transfer_amount_eur() (variable LOW_RISK_TRANSFER_AMOUNT, défaut 100 EUR) — seuil global, pas le seuil par segment 5E.
  • same_owner is True
  • device_trust_level ∈ {HIGH, TRUSTED}
  • intelligence présente et _recent_auth_within(intelligence, low_risk_recent_auth_seconds()) avec LOW_RISK_RECENT_AUTH_SECONDS (défaut 900) — global.
  • Pas de should_require_step_up ni should_force_reauth pour ce tier.
  • risk_score < 25.0 (score déjà calculé sur la somme complète des facteurs).
Example

Si toutes les conditions sont remplies, un facteur downgraded_to_internal_transfer_low (poids 0) est ajouté et final_action_key vaut internal_transfer_low. Un log continuous_auth.risk_downgrade_applied est émis.

Important : les seuils dynamiques par segment (resolve_low_risk_transfer_amount_eur_for_segment) servent la friction adaptative dans continuous_auth_engine, pas cette fonction de downgrade du risk_engine — éviter la confusion entre les deux.

Overrides durs (_apply_hard_behavioral_overrides)

  • Si geo_velocity_raw >= 40 → behavioral_force_reauth, flag geo_impossible_travel.
  • Nouvel appareil + montant > 10 000 EUR + action retrait/transfert → behavioral_force_reauth, new_device_high_amount.
  • Rafale > 5 + action « haute valeur » (déf. _is_high_value_action) → behavioral_force_step_up, burst_high_value.

9. Affinements 5D.1

_device_consistency_factor : device_known (dans liste), device_new, device_no_baseline (empreinte sans liste), ou signal absent avec message explicite ; bonus device_multi_new si ≥3 nouveaux appareils.

_behavioral_action_burst : si types d’actions récents permettent, distinction action_burst_homogeneous vs action_burst_mixed ; plafond via action_burst_homogeneous_cap si n > 5.

_geo_stability_bonus_factor : bonus geo_stability_bonus (poids défaut −5) si pas d’anomalie de vélocité, pays stable, fenêtre temporelle cohérente ou intelligence alignée.

10. Adaptive Intelligence Phase 5E

Activé par ADAPTIVE_INTELLIGENCE_ENABLED. derive_user_segment : ordre — (1) anomalies historiques ≥ 2 → risky_user ; (2) âge < 7 j → new_user ; (3) volume ≥ 50 000 EUR → high_value_user ; (4) âge ≥ 90 j, pas d’anomalies, KYC « trusted » → trusted_user ; sinon normal_user.

extract_segmentation_inputs : âge depuis created_at utilisateur, en-têtes x-user-kyc-level, x-user-lifetime-volume-eur, x-user-successful-actions-count, x-user-historical-anomaly-count (ou champ intelligence).

Ajustement de score : facteur user_segment_adjustment avec poids segment_adjustment_* (défauts dans DEFAULT_RISK_WEIGHTS). Les poids négatifs sont assouplis à 0 si _segment_soften_blocked (reauth stricte, vélocité geo raw ≥ 40, ou niveau préliminaire critique).

Seuils dynamiques : build_dynamic_thresholds_dict expose montant / récence / device_tolerance par segment pour observabilité ; la friction adaptative côté continuous_auth_engine utilise resolve_low_risk_*_for_segment lorsque 5E est actif.

11. Calibration & Learning Phase 5F

  • risk_config.py : DEFAULT_RISK_WEIGHTS, surcharge RISK_WEIGHTS_JSON, scalaires RISK_WEIGHT_<KEY>, calibration_version() via RISK_CALIBRATION_VERSION.
  • risk_experiments.py : assign_variant(user_id, experiment_id) déterministe (SHA-256) ; ratio contrôle RISK_EXPERIMENT_CONTROL_RATIO_PCT ; JSON RISK_EXPERIMENT_<EXP>_VARIANT_A_WEIGHTS_JSON ou RISK_EXPERIMENT_VARIANT_A_WEIGHTS_JSON.
  • runtime_weight_overrides : ContextVar — appliqué dans evaluate_request_risk pour la variante expérimentale.
  • risk_feedback.py : types fraud_confirmed, fraud_suspected, false_positive, successful_action, manual_override ; log risk_feedback.recorded avec user_id_hash opaque.
  • risk_calibration.compute_calibration_suggestions : suggestions uniquement (seuils d’échantillons, pas d’application auto).
Product implication

Toute modification de poids en production relève de la gouvernance (env / release), pas d’apprentissage automatique dans l’API.

14. Exemples détaillés (illustratifs)

Les valeurs numériques suivent les poids par défaut (risk_config.DEFAULT_RISK_WEIGHTS) et supposent les flags nécessaires activés. Les cas réels somment tous les facteurs applicables.

14.1 Trusted user + petit transfert même titulaire (downgrade possible)

Inputs : wallet_transfer, montant 50 EUR, same_owner=true, device TRUSTED, step-up récent (< 900 s), pas de SI step-up/reauth, score final < 25 après somme, ADAPTIVE_FRICTION_ENABLED=true.

Résultat : final_action_key=internal_transfer_low, facteur downgraded_to_internal_transfer_low, log downgrade.

14.2 New user + nouveau device + gros montant

Inputs : segment new_user (âge < 7j) si 5E ; device_new + poids positif ; montant élevé → contributions amount_* positives ; ajustement segment +10.

Résultat : score élevé, niveau souvent high ou critical selon somme ; recommended_outcome step_up ou reauth.

14.3 « Dubai → Paris en 20 minutes »

Inputs : pays courant ≠ pays précédent, delta_seconds = 1200 ≤ 1800 → branche vélocité « 30 min », poids défaut 40 sur le facteur geo_velocity_anomaly ; geo_velocity_raw ≥ 40 → override behavioral_force_reauth.

Résultat : recommended_outcome=reauth, risk_level=critical, flag geo_impossible_travel.

14.4 view_sensitive_data sur device trusted

Base 30 + ajustements device (−15) + fraîcheur selon step-up + facteur lecture read_scope_identity_kyc (+10) + facteurs comportementaux si activés.

14.5 Segment risky_user

Au moins 2 anomalies historiques → segment risky_user ; seuils 40/65 ; ajustement +20 si non bloqué → friction accrue.

14.6 Burst homogène

Si ≥3 actions du même type en 5 min et compteur élevé → code action_burst_homogeneous avec poids défaut 20 (borné).

14.7 Account age faible

account_age_risk avec poids account_age_lt1d (30) ou <7j (20) selon âge en jours.

15. Guide d’interprétation

  • risk_score : somme clampée 0–100 ; comparer aux seuils effectifs (globaux ou segment).
  • risk_level : dérivé des seuils ; peut être forcé critical par override reauth comportemental.
  • factor codes : liste explicable pour audit ; chaque code a un poids dans la somme (sauf cas particuliers documentés).
  • Calibration suggestions : propositions à valider humainnement ; comparer suggested_weight à get_risk_weight.
  • False positive : feedback false_positive + codes de facteurs alimentent les suggestions de baisse de poids si taux suffisant.

17. Limites connues

  • Pas de ML dans le moteur — règles et poids explicites.
  • Dépendance aux en-têtes / champs optionnels : dégradation gracieuse avec poids nuls ou pénalités légères.
  • Compteurs « burst » côté client ou headers — pas de compteur serveur central décrit dans le moteur.
  • Dashboard produit : stockage en mémoire par processus (risk_dashboard_store.py).
  • Calibration : suggestions seulement ; application manuelle des poids.

18. Feuille de route possible

Points évoqués dans les rapports / commentaires du dépôt : affinage géographique, compteurs serveur pour rafales, empreinte enrichie, calibration par segment, décroissance temporelle des scores, scoring qualité avancé — non implémentés comme tels dans les extraits de code consultés ; listés ici comme orientation.
