Conformité & Risk Engine

Décision de risque déterministe, traçable et gouvernée — du scoring aux signaux comportementaux, jusqu’au pilotage produit.

Basé sur risk_engine.py, risk_config.py, rapports Phase 5B–5F et dashboard produit. Aucune promesse réglementaire : diligence technique et exploitabilité audit.

Enjeu réglementaire & audit

Pourquoi le « risque » doit être explicable

Les autorités et partenaires attendent des institutions financières qu’elles démontrent des contrôles proportionnés : savoir pourquoi une action a été bloquée, escaladée ou autorisée.

Un modèle boîte noire pur complique la défense devant un régulateur ou un litige client. Notre approche privilégie un moteur déterministe : scores, seuils, facteurs nommés, logs structurés — complété par un pipeline SIEM et un feedback terrain pour la calibration.

Positionnement

Risque explicable vs ML opaque

Approche courante « scoring ML »
  • Score peu ou pas expliqué au client interne.
  • Ajustements opaques, risque de biais difficile à auditer.
Arquantix (cœur Phase 5)
  • Somme pondérée de facteurs nommés, clamp [0,100].
  • Poids surchargeables (RISK_WEIGHTS_JSON, env).
  • Événements continuous_auth.risk_evaluated, escalades loguées.

Des extensions ML pourraient s’ajouter en aval ; le socle actuel reste auditable par conception.

Empilement

Où vit le Risk Engine

Le moteur s’exécute après la politique statique (sensitive_action_map) et la friction adaptative (5B). Il ne remplace pas une réauth déjà requise par la couche continue ; il peut augmenter la friction (step-up, reauth) selon le score et les overrides comportementaux.

Intégration : continuous_auth_engine → evaluate_request_risk → fusion dans la décision HTTP.

Phase 5C

Score, niveaux, recommandation

  • Score 0–100 : somme des contributions (base par type d’action + device trust SI + fraîcheur step-up + montant + même titulaire + session intelligence + rafales + lecture sensible + pénalités signaux manquants).
  • Niveaux : low / medium / high / critical (seuils RISK_HIGH_THRESHOLD, RISK_CRITICAL_THRESHOLD — ajustables par segment en Phase 5E).
  • Recommandation : allow / step_up / reauth — avec règles explicites (ex. critical → reauth).

Phase 5D

Signaux comportementaux & règles dures

Facteurs optionnels : vélocité géographique, risque pays (carte statique), cohérence appareil, rafales d’actions, multi-sessions, âge de compte, méthode de login. Chaque contribution comportementale est bornée avant sommation.

Overrides durs (exemples documentés) : vélocité extrême → réauth forcée ; nouveau device + gros montant sur transfert → réauth ; rafale + action à forte valeur → step-up forcé.

Phase 5E

Segmentation prudente

Segments déterministes : new_user, normal_user, trusted_user, high_value_user, risky_user — dérivés d’âge de compte, volume, anomalies historiques, KYC.

Effets : seuils high/critical par segment, ajustement de score pondéré, seuils de « bas risque » pour la friction (montants, récence) — avec garde-fous pour ne pas adoucir un contexte déjà dangereux (vélocité geo extrême, etc.).

Phase 5F

Calibration, feedback, expériences

  • Feedback typé (fraude suspectée / confirmée, faux positif, succès, override manuel) avec codes de facteurs — journalisé et alimentant le store du dashboard.
  • Suggestions de calibration : règles fixes sur agrégats (compute_calibration_suggestions) — jamais d’application automatique des poids en production.
  • A/B : variantes déterministes (hash utilisateur + experiment_id), surcharges de poids pour variant_a uniquement.
  • Version : RISK_CALIBRATION_VERSION pour la traçabilité des releases config.

Reclassement contrôlé

Transferts « bas risque»

Sous conditions strictes (friction adaptative activée, même titulaire, montant sous seuil global, appareil trusted, step-up récent, score < 25, pas d’exigence SI contradictoire), un transfert peut être reclassé en internal_transfer_low avec facteur documenté — traçable via log continuous_auth.risk_downgrade_applied.

Dashboard produit

Piloter sans boîte noire

Endpoints admin agrégés (fenêtre glissante) : taux allow / step_up / reauth, distribution des niveaux, facteurs les plus fréquents, corrélation feedback fraude / faux positifs par facteur, métriques par segment et par variante d’expérience, alertes seuils (reauth, anomalies d’overrides durs, taux de feedback fraude), décisions récentes, suggestions de calibration.

Stockage en mémoire par processus API — vue complète en cluster nécessiterait une couche d’agrégation (feuille de route).

Limites assumées

Transparence investisseur

  • Pas de ML dans le cœur du score — la précision repose sur la qualité des signaux (headers, SI, clients).
  • Dashboard MVP : pas d’historique long multi-région sans agrégation centralisée.
  • Funnel « abandon après step-up » non modélisé nativement dans le dashboard.

Roadmap crédible

Évolutions possibles

  • Agrégation Redis / entrepôt pour métriques globales et rétention.
  • Enrichissement des signaux device / géo côté serveur.
  • Calibration par segment avec validation humaine systématique.
  • Couches ML en scoring secondaire, avec garde-fous et explicabilité contratuelle.

Dashboard produit (extraits)

Indicateurs livrés

Summary : taille d’échantillon, score moyen, distribution low→critical, taux allow / step_up / reauth, taux d’évaluations avec drapeaux d’overrides comportementaux durs.

Factors : facteurs les plus fréquents ; corrélation feedback fraude et faux positifs par code de facteur.

Segments & expériences : comparaison par segment utilisateur et par variante A/B (dont critical_level_rate).

Gouvernance des changements

Workflow recommandé

  1. Observer le dashboard et les logs sur une fenêtre représentative.
  2. Examiner les suggestions de calibration (non auto-appliquées).
  3. Valider produit + sécurité une modification de RISK_WEIGHTS_JSON ou seuils.
  4. Déployer avec version RISK_CALIBRATION_VERSION incrémentée.
  5. Contrôler post-release via les mêmes KPIs.
