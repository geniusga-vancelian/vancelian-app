Sécurité & authentification

Couche de confiance continue : sessions serveur, intelligence par session,
authentification continue des actions sensibles, audit & SIEM.

Document aligné sur l’implémentation Arquantix (API FastAPI, mobile Flutter). Comparaisons marché formulées comme objectifs de maturité, non comme études tierces.

Pourquoi investir dans cette couche

La confiance est un actif produit

Les applications financières sont jugées sur leur capacité à protéger l’identité, les fonds et les données réglementées. Une authentification « login + JWT » suffit rarement : le risque évolue pendant la session (changement d’appareil, de pays, de contexte).

Ce que nous avons construit

Une architecture en couches empilées : politiques par type d’action, état de session enrichi, décisions par requête, réponses structurées pour le client, événements d’audit exploitables par un SIEM.

Niveau de maturité vs pratique courante

Où nous nous positionnons

Pratique courante (fintech / apps)
  • Session longue après login, contrôle d’accès grossier.
  • 2FA surtout à l’onboarding ou au login.
  • Peu de corrélation entre device, géo et action sensible.
Arquantix (implémenté)
  • Sessions serveur liées au JWT (sid), refresh rotatif.
  • Intelligence par session + auth continue sur actions sensibles.
  • Passkeys, OTP, biométrie locale reliés aux politiques.
  • Événements normalisés vers SIEM (Datadog / OpenSearch).

La profondeur réelle dépend du déploiement (feature flags activés, routes branchées, clients à jour).

Login vs protection continue

Deux temporalités distinctes

Authentification initiale : mot de passe, passkey login, OTP orchestré à l’issue de session — établit AuthSession et tokens.

Protection continue : chaque opération sensible (virement, données KYC, révocation de sessions, etc.) peut exiger une preuve fraîche (OTP, passkey) ou une réauth complète selon le contexte.

Les flux non authentifiés (login, démarrage passkey) ne passent pas par require_continuous_auth_for_action : ils n’ont pas encore de sid de session applicative.

2FA & accès renforcé

Passkeys, OTP, biométrie

  • Passkeys (WebAuthn) : enregistrement et révocation protégés par des politiques (security_settings_change) ; liste des credentials par view_sensitive_data.
  • OTP / step-up : peut être exigé au login (orchestrateur adaptatif) et surtout sur action sensible via le moteur d’auth continue.
  • Biométrie / code local : signal client (relock, hints JWT) aligné avec should_relock_local pour les tiers HIGH.

Session Intelligence

État vivant par session

La table auth_session_intelligence agrège : force d’auth, niveaux de confiance, scores de risque, activité, IP/pays, flags step-up / relock, last_step_up_at, raisons structurées.

Un middleware optionnel met à jour l’activité sur les requêtes authentifiées (JWT avec sid), hors chemins login/refresh bruts.

Authentification continue

Une dépendance, des réponses machine-lisibles

require_continuous_auth_for_action("clé") : si les flags globaux sont actifs et qu’une intelligence existe, la requête est évaluée.

Réponses HTTP structurées : session.reauth_required (401), session.step_up_required (403), avec next_step (otp_or_passkey, full_reauth, etc.), reason_codes, extrait de policy.

Couche UX : ux_message, ux_tone, ux_context — le mobile affiche un message court non technique.

Friction adaptative (Phase 5B)

Réduire la friction sans abandonner la sécurité

Lorsque les flags le permettent, des règles déterministes peuvent alléger le step-up pour certains transferts (montant faible, appareil de confiance, step-up récent) ou certaines lectures sensibles — sans contourner une réauth déjà exigée par la politique stricte ou la session.

Cette couche vit dans le continuous_auth_engine, au-dessus de la carte d’actions et en dessous du moteur de risque optionnel.

Zero Trust ciblée

Révocation & liste des sessions

La révocation globale des sessions combine une politique dédiée (session_revoke_all) et des garde-fous zero-trust sur la route — l’utilisateur doit prouver qu’il contrôle encore la session courante avant d’invalider les autres.

La consultation des sessions actives est traitée comme une lecture sensible (view_sensitive_data) avec audit métier.

Cartographie des risques

Actions sensibles centralisées

Une seule source de politique (ACTION_POLICIES) : retraits, transferts, bénéficiaires, paramètres sécurité, données sensibles, révocation globale des sessions, etc.

Évite la prolifération de garde-fous incohérents ; les équipes branchent les routes sur une clé stable pour l’audit.

Observabilité

SIEM & événements

Pipeline documenté : persistance, anonymisation, export Datadog ou OpenSearch/ELK, corrélation légère, alertes critiques (webhook), rétention paramétrable.

Préfixes typiques : auth.session.*, sensitive_action.* — utiles pour détection d’incident et preuve de diligence.

Client mobile

Alignement Flutter

Parsing des erreurs : sensitive_action_http.dart — codes, next_step, champs UX.

Gestion du relock / biométrie : SessionIntelligenceManager et politique locale cohérente avec les claims JWT.

Gouvernance & déploiement

Feature flags

Rollout progressif documenté : activer d’abord SESSION_INTELLIGENCE_ENABLED, puis CONTINUOUS_AUTH_ENABLED, puis brancher route par route.

Si désactivé, le comportement historique est préservé — important pour la continuité d’exploitation.

Limites & transparence

Ce qu’il reste à renforcer

  • Certaines routes « moteur portfolio » historiques utilisent un contexte acteur sans la même chaîne JWT + auth continue — dette connue, plan de convergence.
  • API « step-up completed » côté serveur pour ancrer last_step_up_at : piste d’évolution.
  • Coverage client : intercepteur HTTP global recommandé pour uniformiser les parcours step-up.

JWT enrichi

Claims de confiance dans le token

Les tokens d’accès peuvent porter : sid (session), dtrust (device trust), auth_str, strust (session trust), lstup (dernier step-up), relock, bio_req — le client Flutter persiste un snapshot pour aligner biométrie et relock local avec le serveur.

Qualité logicielle

Tests & non-régression

Le dépôt inclut des suites ciblées (session intelligence, auth continue, transferts sensibles, custody, etc.) pour sécuriser les évolutions — prérequis pour une vélocité de release sans casser les garde-fous.

Synthèse investisseur

Message clé

Arquantix ne se contente pas d’authentifier une fois : elle mesure et réagit au risque à chaque action sensible, avec des politiques versionnables, des journaux structurés et une expérience utilisateur guidée par des réponses API explicites — socle crédible pour une plateforme réglementée et compétitive sur la sécurité.
