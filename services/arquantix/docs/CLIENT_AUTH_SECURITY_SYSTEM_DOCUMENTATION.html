1. Introduction

L’objectif global est de protéger le client au-delà du simple mot de passe initial : la plateforme distingue l’authentification initiale (login, passkey login, refresh) de la protection continue des actions sensibles (virements, retraits, paramètres de sécurité, données d’identité, etc.).

Key takeaway

La philosophie documentée dans le code est celle d’une protection progressive : session serveur (AuthSession) liée au JWT via sid, enrichissement par session intelligence (état de risque par session), puis continuous auth qui applique les politiques définies dans sensitive_action_map. Les escalades step-up (OTP / passkey) et reauth (nouvelle authentification complète) sont des réponses API structurées consommées par le mobile.

Différence fondamentale : le login et les flux de pré-auth (ex. démarrage passkey login sans session utilisateur établie) ne passent pas par require_continuous_auth_for_action, car cette dépendance suppose un JWT utilisateur avec session liée — voir section 3.

3. Système de login

3.1 Parcours authentifié vs pré-auth

Le flux principal de session passe par services/auth/refresh_session.py : création ou renouvellement de AuthSession, émission access + refresh JWT. Lorsque SESSION_INTELLIGENCE_ENABLED est actif, initialize_session_intelligence crée la ligne auth_session_intelligence à l’issue d’une session réussie.

Les endpoints de login non authentifiés (ex. POST /auth/login, POST /auth/passkeys/login/start et .../finish) ne portent pas de Bearer utilisateur avec sid : ils ne sont pas concernés par require_continuous_auth_for_action.

3.2 Pourquoi le continuous auth n’englobe pas le login

La dépendance require_continuous_auth_for_action décode le JWT, exige un claim sid, charge AuthSession valide et la ligne intelligence. Sans sid, le code laisse passer l’utilisateur (comportement de rétrocompatibilité documenté dans le docstring du module). Les flux login / pré-auth ne satisfont pas ce prérequis par définition.

3.3 JWT, sid et cycle de vie

  • create_access_token (auth.py) peut inclure sid, dtrust, auth_str, strust, lstup, relock, bio_req, etc.
  • La session « stable » côté serveur est AuthSession (refresh rotatif, révocation, expiration).
  • Les jetons courts type create_registration_otp_token (inscription 2FA) n’ont pas de sid de session applicative complète au sens auth_sessions.

4. Passkeys, OTP, 2FA et périmètre

Les routes passkeys vivent sous /auth/passkeys (passkeys_routes.py) :

  • Login passkey (login/start, login/finish) : pas de require_continuous_auth_for_action — établissent une session comme le login classique.
  • Enregistrement / révocation : protégés par security_settings_change (register start/finish, revoke).
  • Liste des passkeys (GET /auth/passkeys) : protégée par view_sensitive_data + enregistrement d’événement métier sensitive_action.completed en lecture seule.

Le step-up au login (OTP requis par l’orchestrateur adaptatif, flags sur AuthSession.step_up_otp_required) est une couche distincte du step-up sur action sensible décidé par continuous_auth_engine après coup.

Key takeaway

Les clés api_key_create et data_export existent dans ACTION_POLICIES mais des rapports d’audit (PHASE_FINAL_API_KEYS_SENSITIVE_DATA_REPORT.md) indiquent l’absence de routes métier dédiées correspondantes à ce jour — la politique est prête pour un branchement futur.

5. Continuous Auth

Définition : décision par requête pour une action_key sensible, à partir de la politique (sensitive_action_map) et de l’état AuthSessionIntelligence (et du moteur de risque si activé).

Pourquoi : un token valide ne suffit pas si le contexte de session (risque, appareil, pays, etc.) exige une preuve fraîche d’identité.

5.1 Fonctionnement de require_continuous_auth_for_action

Si CONTINUOUS_AUTH_ENABLED ou SESSION_INTELLIGENCE_ENABLED est faux → retour immédiat de l’utilisateur sans blocage. Sinon, évaluation via evaluate_request_security_context ; si dec.allow est faux, levée d’HTTPException avec detail structuré.

5.2 Codes HTTP et champs detail

Statutdetail["code"]Condition

Le payload inclut notamment : message, action_key, reason_codes, next_step (via next_step_hint), policy (extrait de SensitiveActionPolicy), et si le moteur de risque a tourné : risk_score, risk_level, risk_factors, recommended_outcome, éventuellement final_action_key.

5.3 Valeurs de next_step (next_step_hint)

  • full_reauth — reauth complète
  • otp_or_passkey_then_biometric — step-up + biométrie
  • biometric_or_passcode — biométrie / code local
  • otp_or_passkey — OTP ou passkey
  • none — aucune indication machine additionnelle

6. Actions sensibles

sensitive_action_map.py définit ACTION_POLICIES : chaque entrée est une SensitiveActionPolicy avec required_auth_level, requires_step_up, requires_recent_auth_seconds, requires_biometric, allowed_if_device_trusted_only.

Protection d’une route : ajouter Depends(require_continuous_auth_for_action("<action_key>")) en plus des garde-fous métier (ownership, zero-trust sur certaines routes, etc.). Les handlers appelant record_sensitive_action_completed / failed enregistrent le résultat après exécution métier — une requête bloquée par la dépendance ne produit pas d’événement sensitive_action.completed côté métier (les audits sensitive_action.requested / step-up / reauth sont émis dans la dépendance).

Exemples de distinction : view_sensitive_data (accès données identité / KYC / listings sensibles) vs withdrawal / wallet_transfer (mouvements) vs beneficiary_add vs security_settings_change vs session_revoke_all (politique HIGH mais requires_recent_auth_seconds absent pour cette clé — la friction repose sur les autres règles moteur/SI).

Pour les transferts, des en-têtes optionnels sont lus par la dépendance : X-Transfer-Amount-Eur, X-Transfer-Same-Owner, X-Recent-Similar-Actions (friction adaptative / risque).

7. Session intelligence

Définition : état persistant par session dans auth_session_intelligence, mis à jour lors des refresh/requêtes (service + middleware d’activité).

Signaux pris en compte (d’après session_intelligence_service.py et evaluate_session_risk) : scores globaux utilisateur / fraude, raisons telles que ip_changed, country_changed, fingerprint_changed, niveau de confiance appareil (device_trust_level), niveau de confiance de session dérivé (compute_session_trust), step_up_required, last_step_up_at, relock_required.

Influence sur l’auth continue : should_require_step_up et should_force_reauth utilisent le tier d’action et le score / raisons ; should_relock_local alimente l’exigence biométrique locale pour certains tiers HIGH sans passkey dans le jeton.

8. Device trust et recent auth

Dans le moteur, un appareil est considéré comme suffisamment fiable pour les heuristiques « trusted » si device_trust_level est HIGH ou TRUSTED (_device_trusted_enough).

Recent auth : last_step_up_at est comparé aux fenêtres requires_recent_auth_seconds de la politique (et aux seuils « low risk » si friction adaptative active).

La politique view_sensitive_data combine trusted_only et fenêtre récente (600 s dans la carte actuelle) : lecture sensible renforcée côté règles.

9. UX et expérience client

Le module continuous_auth_ux.py enrichit le detail sans modifier les décisions de sécurité. Champs :

  • ux_message — texte court non technique
  • ux_tone — soft | warning | critical (peut être rehaussé si risk_level critique / élevé)
  • ux_action_label — libellé de CTA suggéré
  • ux_context — withdrawal | security_change | data_access

La priorité des reason_codes pour choisir un message unique est définie dans _REASON_PRIORITY. Côté Flutter, SensitiveActionErrorShape.displayMessage préfère ux_message au message API brut.

10. Friction adaptative (Phase 5B)

Activée par ADAPTIVE_FRICTION_ENABLED (security_env.py). Le moteur peut réduire la friction pour :

  • wallet_transfer — montant sous seuil EUR, appareil trusted, step-up récent, absence d’exigence SI contradictoire → raison adaptive_low_friction_transfer
  • view_sensitive_data en MEDIUM — device trusted + récence + pas d’imposition step-up par SI → adaptive_low_friction_view_sensitive
Security note

Une reauth stricte déjà posée (require_reauth) n’est pas contournée par le moteur de risque (_merge_risk_into_decision dans continuous_auth_engine.py). Le moteur de risque (Phase 5C) peut encore augmenter la friction (step-up / reauth) mais respecte les refus durs préexistants.

11. Exemples de flux bout en bout

Les scénarios ci-dessous décrivent le comportement lorsque les flags sont actifs et qu’une ligne intelligence existe. Sinon, la dépendance est transparente.

11.1 Lecture d’identité sensible

État initial : JWT avec sid, intelligence présente, device non trusted ou last_step_up_at absent / trop ancien pour la fenêtre.

Appel : route protégée par view_sensitive_data (ex. identité portfolio selon route).

Décision : 403, session.step_up_required, reason_codes incluant souvent device_not_trusted et/ou recent_auth_required.

Client : interpréter next_step et afficher ux_message.

11.2 Changement de paramètre sécurité

Routes passkeys register/revoke : security_settings_change. Step-up fréquent car politique HIGH + fenêtres récentes + biométrie possible selon contexte.

11.3 Listing passkeys

GET /auth/passkeys — view_sensitive_data ; succès → sensitive_action.completed avec métadonnée lecture seule.

11.4 Révocation globale des sessions

POST /auth/revoke-all : session_revoke_all + enforce_zero_trust_or_raise dans main.py (hors mode test). Succès → sensitive_action.completed.

11.5 Device non trusted

Pour les actions avec allowed_if_device_trusted_only, l’absence de confiance HIGH/TRUSTED déclenche device_not_trusted et step-up.

11.6 Recent auth trop ancien

recent_auth_required lorsque la politique définit requires_recent_auth_seconds et que last_step_up_at est hors fenêtre.

11.7 Reauth complète

Ex. score ou raisons déclenchant should_force_reauth (pays changé + tier medium/high, ou score ≥ 85 selon implémentation actuelle) → 401 session.reauth_required, audit sensitive_action.reauth_required.

12. Audit, hooks et observabilité

Service : services/auth/security_events_service.py (activation typique AUTH_SECURITY_EVENTS_ENABLED).

Dans la dépendance require_continuous_auth_for_action : sensitive_action.requested au début ; puis selon cas auth.session.reauth.triggered, sensitive_action.reauth_required, auth.session.step_up.triggered, sensitive_action.step_up_required, ou sensitive_action.blocked.

Dans les handlers métier (après succès ou échec logique) : sensitive_action.completed / sensitive_action.failed via sensitive_action_events.py.

Intérêt : corrélation SIEM, preuves pour conformité, diagnostic support (distinction « bloqué avant handler » vs « échec métier après »).

13. Sécurité côté routes (aperçu)

La dépendance est branchée notamment sur : main.py (revoke-all, liste sessions), passkeys_routes.py, security_admin_routes.py, persons/routes.py, routes custody, portfolio_engine (clients, execution, orders, settlement, trades, orchestrator). Référence grep require_continuous_auth_for_action pour la liste exhaustive à jour.

Security note

Le rapport PHASE_4A_PORTFOLIO_ENGINE_AUTH_ALIGNMENT_REPORT.md documente une protection inégale : les routes qui ne reposent que sur ActorContext (en-têtes acteur portfolio) sans JWT utilisateur ne bénéficient pas de l’auth continue telle que décrite ici. C’est une dépendance structurelle connue, pas un oubli ponctuel dans un seul fichier.

15. Limites connues / points d’attention

  • ActorContext-only : voir section 13 et le rapport Phase 4A.
  • Legacy : JWT sans sid — pas de blocage par la dépendance ; risque de divergence avec l’objectif Zero Trust si clients anciens.
  • Mobile hors FastAPI : toute logique purement locale (relock, biométrie) complète mais ne remplace pas les réponses serveur.
  • Qualité des signaux : SI dépend des mises à jour (IP, pays, empreinte) et de l’absence d’erreurs de configuration.
  • Gaps documentés dans SESSION_INTELLIGENCE_CONTINUOUS_AUTH_REPORT.md : branchements métier additionnels, échantillonnage SIEM, API « step-up completed » pour poser last_step_up_at — à traiter selon roadmap.

16. Guide pratique pour les équipes

Backend

  1. Choisir ou ajouter une action_key dans ACTION_POLICIES si besoin.
  2. Ajouter Depends(require_continuous_auth_for_action("ma_cle")) à la route, avec get_current_user / auth existante.
  3. Appeler record_sensitive_action_completed / failed selon le résultat métier.
  4. Pour les transferts, documenter les en-têtes attendus par le client si la friction adaptative / risque doit les utiliser.

Mobile

Utiliser parseSensitiveActionError ; guider l’utilisateur avec displayMessage ; router vers OTP / passkey / re-login selon next_step et code.

Support

Distinguer 401 reauth (session globalement insuffisante) vs 403 step-up (preuve fraîche). Croiser avec événements sensitive_action.* et métadonnées reason_codes.

Produit

La friction est la conjonction politique (sensitive_action_map) et contexte (device_trust_level, scores, fenêtres temps). Les textes UX sont ajustables côté continuous_auth_ux.py sans changer les règles dans le moteur.

ReauthNouvelle authentification complète de session (souvent 401 session.reauth_required).

Recent authPreuve récente via last_step_up_at dans les fenêtres définies.
sidClaim JWT : identifiant de AuthSession.
action_keyClé stable dans ACTION_POLICIES (audit, SIEM).
Risk levelNiveau du moteur de risque Phase 5C (low / medium / high / critical) lorsque le moteur est activé.
Continuous authDécision par requête sur action sensible, après login.
