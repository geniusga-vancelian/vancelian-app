1. Introduction

Objectif du dashboard : offrir une visibilité agrégée sur les évaluations du moteur de risque (scores, niveaux, outcomes recommandés, segments, expériences) et sur les feedbacks terrain, afin de piloter friction, sécurité et calibration sans analyser uniquement des logs bruts.

Pourquoi il existe : après le déploiement des phases 5C–5F, les équipes ont besoin d’un tableau de bord unique pour suivre l’effet produit du moteur (taux d’allow / step_up / reauth), la distribution des niveaux de risque, les facteurs les plus présents, et les signaux d’alerte simples.

À qui il s’adresse : produit, ops, sécurité, management, support avancé — chacun avec un angle différent (friction UX, incidents, conformité, synthèse, investigation cas par cas).

Ce qu’il permet de décider : ajuster des seuils ou poids (via processus hors dashboard : variables d’environnement, RISK_WEIGHTS_JSON, etc.), lancer ou arrêter une expérience A/B, prioriser une analyse de facteurs trop frictionnants ou trop liés à la fraude, et valider une release en comparant avant/après sur une fenêtre donnée.

What this means

Les chiffres reflètent le trafic vu par ce worker API. En cluster multi-instances, chaque instance a son propre buffer mémoire : la vue n’est pas globale tant qu’une couche d’agrégation centralisée n’existe pas.

Risk engine → record_* → Store (deque) → aggregate_* → API → Proxy Next → UI

4. Résumé /summary

ChampDéfinition (code)
window_hoursFenêtre glissante en heures.
sample_sizeNombre d’évaluations dans la fenêtre.
avg_risk_scoreMoyenne des risk_score ; null si échantillon vide.
distributionComptes par risk_level : low, medium, high, critical.
allow_ratePart des recommended_outcome == "allow".
step_up_ratePart des "step_up".
reauth_ratePart des "reauth".
anomaly_detection_ratePart des évaluations où behavioral_anomaly est vrai. Dans le code, l’enregistrement reçoit behavioral_flags=list(beh_flags) avec beh_flags issu uniquement des overrides durs (_apply_hard_behavioral_overrides : ex. geo_impossible_travel, new_device_high_amount, burst_high_value) — ce n’est pas la présence de facteurs comportementaux généraux (géo, rafale, etc.) dans le score.
What this means

Les taux d’outcome décrivent ce que le moteur a recommandé après évaluation, pas le résultat métier final côté utilisateur (pas de funnel « abandon après step-up » dans ce MVP).

Cas d’interprétation

  • allow très élevé : beaucoup de parcours classés « allow » au niveau risque — friction faible côté moteur ; à croiser avec la sécurité (volume, segments).
  • reauth qui monte : hausse des recommandations de ré-authentification complète — peut indiquer scores élevés, contexte SI strict, ou overrides comportementaux.
  • anomaly rate qui explose : hausse des cas où un override dur a posé un drapeau (beh_flags) — typiquement vélocité géographique extrême, nouveau device + gros montant, ou rafale + action à forte valeur ; croiser avec /recent et les logs métier.

5. Facteurs /factors

  • top_factors_by_frequency : facteurs les plus souvent présents dans les listes de facteurs des évaluations (comptage par occurrence de code).
  • top_factors_in_fraud_feedback : codes parmi les feedbacks fraud_confirmed / fraud_suspected.
  • top_factors_in_false_positive_feedback : codes pour false_positive.
  • fraud_feedback_rate : part des événements feedback de type fraude parmi tous les feedbacks de la fenêtre (pas parmi les évaluations).
  • false_positive_rate : idem pour les faux positifs.
What to do

Un facteur fréquent n’est pas forcément « mauvais » : il peut être central au modèle. Le comparer aux listes fraude et faux positifs : si le même code domine en FP, envisager de baisser son poids (processus calibration) ; s’il domine en fraude, envisager de le renforcer.

Common mistake

Confondre fraud_feedback_rate avec un taux de fraude dans la population utilisateurs : c’est un taux parmi les feedbacks enregistrés, dont le volume peut être faible.

6. Segments /segments

Réponse : by_segment — pour chaque clé de segment (new_user, normal_user, trusted_user, high_value_user, risky_user, etc.), sample_size, avg_risk_score, allow_rate, step_up_rate, reauth_rate.

Step_up_rate élevé sur trusted_user : anormal si l’attente produit est une friction minimale pour ce segment — vérifier évolution des poids segment, seuils dynamiques, ou biais d’échantillon (peu de lignes).

What this means

Les segments viennent des évaluations enregistrées (user_segment calculé côté moteur à l’évaluation). Comparer les colonnes entre segments sur la même fenêtre pour le tuning.

7. Expériences /experiments

Une ligne apparaît si des évaluations ont un experiment_id non vide (sinon liste vide — message UI : pas d’évaluation avec RISK_EXPERIMENT_ID).

Champs : experiment_id, variant, sample_size, allow_rate, step_up_rate, reauth_rate, avg_risk_score, critical_level_rate (part de risk_level == "critical").

What to do

Comparer deux variantes du même experiment_id : si une variante augmente allow_rate mais aussi critical_level_rate ou les feedbacks fraude (via /factors), le trade-off est défavorable côté sécurité — décision collective produit / sécurité.

8. Alertes /alerts

Variables d’environnement (défauts entre parenthèses) :

  • RISK_DASHBOARD_ALERT_REAUTH_RATE_GT (0.35)
  • RISK_DASHBOARD_ALERT_ANOMALY_RATE_GT (0.30)
  • RISK_DASHBOARD_ALERT_FRAUD_FEEDBACK_RATE_GT (0.15)

Conditions : pour reauth / anomaly, sample_size ≥ 20 ; pour fraude, feedback_sample_size ≥ 10 et taux ≥ seuil.

severity : warning pour reauth et anomaly ; critical pour spike fraud feedback.

What this means

Une alerte warning signale un dépassement de seuil sur la fenêtre — pas nécessairement un incident de sécurité avéré ; un critical sur feedback fraude mérite une revue prioritaire des retours terrain.

9. Décisions récentes /recent

Liste triée par temps décroissant (jusqu’à limit). Champs : ts, action_key, risk_score, risk_level, recommended_outcome, user_segment, variant, experiment_id, behavioral_anomaly, factor_codes (tronqué à 20 codes).

Investigation : partir d’une ligne suspecte → vérifier le segment, le outcome, les facteurs → croiser avec la politique d’action (sensitive_action_map) et les logs applicatifs si besoin.

10. Suggestions calibration

Le backend reconstruit des objets RiskFeedback à partir des snapshots du store (champs feedback + factor_codes) et appelle compute_calibration_suggestions (risk_calibration.py) — suggestions uniquement, pas d’application automatique des poids.

Réponse : suggestions (liste avec factor_code, current_weight, suggested_weight, confidence, reason), feedback_events_used.

What to do

Workflow : produit / sécurité examinent la suggestion → décision de changer RISK_WEIGHTS_JSON ou équivalent → release documentée → revalider sur le dashboard après déploiement.

12. Guide de lecture métier (si / alors)

ObservationPiste d’action
reauth_rate monteVérifier distribution critical, alertes reauth, facteurs liés à SI / géo ; revue des seuils RISK_* ou contexte session.
fraud_feedback_rate monteAnalyser top_factors_in_fraud_feedback ; ne pas baisser la friction sans preuve ; escalade sécurité.
false_positive_rate monteIdentifier facteurs dans top_factors_in_false_positive_feedback ; envisager baisse de poids ciblée via calibration.
trusted_user trop frictionnéComparer step_up_rate / allow_rate à d’autres segments ; vérifier segment_adjustment et données d’entrée (en-têtes volume, âge).
Variante A/B : plus d’allow mais fraude upNe pas retenir la variante sur le seul allow_rate ; exiger stabilité des feedbacks fraude et du critical_level_rate.
Facteur en tête des FPPrioriser une étude de calibration sur ce code ; valider avec le support / cas réels.

13. Cas d’usage opérationnels

  • Monitoring quotidien : ouvrir 24 h, lire cartes friction + alertes + tableau segments.
  • Tuning hebdo : fenêtre 7 j, comparer segments et facteurs ; noter les suggestions calibration.
  • Investigation incident : /recent + corrélation manuelle avec l’heure et l’action_key.
  • Revue d’expérience : onglet expériences, comparer variantes et critical_level_rate.
  • Validation de release : capture avant / après (même fenêtre) sur les mêmes KPIs.
  • Pré / post changement de config : noter sample_size suffisant avant de conclure.

14. Limites connues

  • Stockage en mémoire, buffers bornés (_MAX_EVAL_EVENTS / _MAX_FEEDBACK_EVENTS).
  • Vue par processus — pas de total cluster.
  • Pas de funnel natif « abandon après step-up ».
  • MVP : pas d’historique long d’alertes ni de tendances stockées.

15. Recommandations d’usage

  • window_hours : 24 h pour le quotidien ; 7 j pour les tendances ; 6 h après un déploiement urgent.
  • Hebdomadaire : allow_rate, reauth_rate, fraud_feedback_rate / false_positive_rate (si feedbacks suffisants), top facteurs.
  • Seuils : ajuster d’abord les alertes dashboard (env) pour limiter le bruit, puis les poids risque avec gouvernance.
  • Coordination : produit (friction), sécurité (seuils et fraude), support (cas clients alimentant les feedbacks).

Limites structurelles — pistes v2

  • Agrégation centralisée (Redis, data warehouse) pour vue cluster et rétention longue.
  • Funnel post–step-up (événements métier + corrélation IDs).
  • Historique d’alertes et courbes de tendance.
  • RBAC fin sur la route admin si besoin de séparer lecture vs configuration.
  • Export CSV / rapport planifié pour compliance.
